## 〇 · v3 重写要点

- **格大小**: 192 → **256**. 192 = 64 × 3 (Z/3 cyclic Shi), 256 = 64 × 4 = (Z/2)⁸ (V₄ Klein Shi).

## 一 · 爻到维 (不变 from v1)

爻不是普通对象, 而是最小可分差异. 它有两种读法:

| 读法 | 形式 |
|---|---|
| 爻位 | 一条可独立翻转 / 判别的二值维度 |
| 爻值 | 该维度上的一个点：阴或阳 |

因此:

```text
一个爻位 = 一维 Bool
一个爻值 = Bool 中的一个点
n 个爻位 = Bool^n = {阴, 阳}^n
```

层级展开为 (R₀..R₈ strict uniform):

| 爻位数 | 结构 | 点数 | 易层读法 |
|---:|---|---:|---|
| 0 | `Unit` | 1 | 一点 / 未分 (R₀ 太极) |
| 1 | `Bool` | 2 | 一爻 / 阴阳 (R₁) |
| 2 | `Bool^2` | 4 | 四象 (R₂) |
| 3 | `Bool^3` | 8 | 八卦 (R₃) |
| 4 | `Bool^4` | 16 | 面 (R₄ Mian) |
| 5 | `Bool^5` | 32 | 五爻 (R₅, provisional) |
| 6 | `Bool^6` | 64 | 六十四卦 (R₆) |
| 7 | `Bool^7` | 128 | 因卦 (R₇ Cell128) |
| **8** | **`Bool^8`** | **256** | **果卦 (R₈ Cell256, V₄ Shi)** |

所以 "从爻到维" 的关键不是 "爻等于 0 维", 而是:

```text
爻位产生维度; 爻值落成点.
```

这也解释为什么三爻自然是八卦, 六爻自然是六十四卦: 每加一个可独立判别的爻位, 就把状态空间乘上一个 `Bool`. **8 爻位之 (Z/2)⁸ = 256 是 R-hierarchy 之自然 8-bit 闭合** (per [yi-RO-hierarchy.md](../docs-next/10_formal_形式/yi-RO-hierarchy.md) §3.8).

## 二 · 物间缘 (不变 from v1)

本文把更底层的成事结构写为:

```text
物 × 间 × 缘 -> 事
```

其中:

| 字 | 读法 |
|---|---|
| 物 | 可被指认、可承载同一性的对象面 |
| 间 | 可定位、分辨、相邻、先后、内外的关系场 |
| 缘 | 使物与间发生牵连并可能成事的条件关系 |
| 事 | 物在间中依缘而成的一次显现 |

这比直接说 `物 × 间 × 事` 更细, 因为 `事` 不再是根槽位, 而是结果层. 现有 `物间事三位一体` 文档仍可保留为接口读法:

```text
WuJianShiExpression Thing Interval Event
```

在本文中, 它被解释为:

```text
Thing    -> 物
Interval -> 间
Event    -> 物间缘合成后的事
```

所以旧接口没有被否定; 只是 `缘` 补上了 "为什么能成事" 的条件层.

## 三 · 名是指针 (不变 from v1)

名不是对象本身. 名更像一个指针、handle 或可审计入口:

```text
名 : 爻 -> Target
Target = 物 | 间 | 缘
```

名可以指向物、间或缘; 类则是名的分束, 不是简单 alias.

```text
类 = 按同指、同谓、同缘, 把名分束
```

构名时, 许多单字不是被命名的对象, 而是构名算子:

| 字 | 构名作用 | 例 |
|---|---|---|
| 者 | 动作的主动承载位 | `持者`、`出者` |
| 所 | 动作的被指 / 被偶位 | `所指`、`所偶` |
| 之 | 属格 / 投影 / 应用 | `爻之名` |
| 为 | 视角转换 / ascription | `爻为物`、`爻为缘` |
| 主 | 单主面 | `主面` |
| 诸 | 多面 / 多系 / 多修 | `诸面`、`诸系`、`诸修` |
| 于 | 时位锚定 | `生于`、`起于` |
| 是 | 谓词化 / 判为某类 | `是物`、`是缘` |

这给出一组待对齐的核心构式:

```text
名        <- 爻
类        <- 别·名 / 分·名
主面      <- 主·面
诸面      <- 诸·面
生于      <- 生·于
焚于      <- 焚·于
起于      <- 起·于
寝于      <- 寝·于
持者      <- 持·者
出者      <- 出·者
所指      <- 所·指
所偶      <- 所·偶
诸系      <- 诸·系
诸修      <- 诸·修
见度      <- 见·度
爻之名    <- 爻·之·名
爻为物    <- 爻·为·物
爻为间    <- 爻·为·间
爻为缘    <- 爻·为·缘
```

其中 `爻为者 / 爻为持 / 爻为物 / 爻为间 / 爻为缘 / 爻为期` 不是六个实体列表, 而是同一爻在不同目标类型下的视角化.

## 四 · 256 的拓扑 (v3 升级)

Lean 中 (per [`Cell256.lean`](../formal/SSBX/Foundation/Bagua/Cell256.lean) §2):

```text
Cell256 = Hexagram × Shi
Hexagram = Bool^6
Shi = V₄ Klein 四群 = {道, 已, 今, 未}
     ≅ (因, 果) ∈ Bool²  (via Shi.toYinGuo 双射)
```

所以:

```text
Cell256 = Bool^6 × V₄
        = Bool^6 × Bool²  (via 因/果 双射)
        = Bool^8 = (Z/2)⁸
```

这个式子有两个方向:

| 方向 | Lean 对象 | 拓扑读法 |
|---|---|---|
| 六爻方向 | `flip1` 至 `flip6` (mask `xooooooo`..`oooooxoo`) | 每个 Shi 状态层内是一张 6-cube |
| Shi V₄ 方向 | `shiCuo / shiZong / shiCuoZong` (V₄ involutions) + `yin / tou` (R₇/R₈ atomic XOR mask) | 每个卦点上挂一个 V₄ Klein 四群 |

若只看六爻翻转边, `Cell256` 分成四个互不相通的 64 点 6-cube:

```text
Q6 × {道}    (64 个 dao-cells, 含 `(qian, dao) = OX["oooooooo"] = origin`)
Q6 × {已}    (64 个 ji-cells)
Q6 × {今}    (64 个 jin-cells)
Q6 × {未}    (64 个 wei-cells)
```

加入 Shi V₄ 四个 toggle (cuo / zong / cuoZong / id) 后, 每个卦点上的四时态由 V₄ Klein 群关联:

```text
道 ←cuo→ 已     (cuo = toggle 因 axis = Shi.cuo)
道 ←zong→ 未    (zong = toggle 果 axis = Shi.zong)
道 ←cuoZong→ 今 (cuoZong = toggle 双 axis = Shi.cuoZong = V₄ central)
已 ←zong→ 今, 未 ←cuo→ 今 (composition)
```

V₄ Cayley graph: 4 顶点 (道/已/今/未), 6 条无向边 (3 generators × 2 = 3 双向 edges per pair; total `C(4,2) = 6` 边 in K_4 形态)。具体 V₄ 之 Cayley graph 是 `K_4` 或 `C_4` 取决于 generator set; here 3 generators (cuo / zong / cuoZong) 给 K_4 (complete) 形态.

于是本文可把 256 的图读作:

```text
Q6 □ V4
```

也就是六维二值立方体与 V₄ Klein 四群 Cayley graph 的 Cartesian product. 这里的 `□` 是本文的图论读法, 不是当前 Lean 中已有的命名定理.

> **v1 之 `Q6 □ C3` (6-cube × Z/3 cyclic graph)** vs **v3 之 `Q6 □ V4` (6-cube × Klein 4-group Cayley graph)**: V₄ 是 abelian Klein 群, 无 cyclic 结构, 但有 identity (= 道) — 这是 v1 → v3 之核心结构升级.

还要区分另一个容易混淆的 256:

| 256 来源 | 公式 | 含义 |
|---|---|---|
| `Cell256` | `64 × 4` | 六十四卦乘 V₄ Shi 4 时态 |
| 8-cube 节点数 | `2^8` | 八维立方体的顶点数 |
| ASCII 字符 | `2^8` | 1 byte 之 cardinality |

三者数值相同, 来源不同. Cell256 之 64 × 4 分解是「卦 × Shi V₄」; 8-cube 是 (Z/2)⁸ 之 Cayley graph 形态; ASCII 是工程系统在 binary 上之 8-bit closure. **R₈ Cell256 = ASCII cardinality** 是 (Z/2)⁸ self-similar closure 之自然结果 (per [Theorem K Corollary K.3](../docs-next/10_formal_形式/yi-calculus-theorem.md))。

## 五 · 371 的位置 (v3 仅 cell axis 升级)

`371` 是文言算子目录的行数:

```text
allOperatorIds.length = 371
```

它不是第 371 个维度, 也不是 371 个状态点. 它更像一套可作用、可读、可索引的操作表.

**v3 关键不变**: 371 算子目录 itself 在 V₄-Shi-migration 中 0 改动. 仅 cell axis 由 Cell192 升级为 Cell256. Lean 形式 in [`OperatorCellMap.lean`](../formal/SSBX/Text/OperatorCellMap.lean):

```lean
abbrev OperatorCell : Type := OperatorId × Cell256

def allOperatorCells : List OperatorCell :=
  allOperatorIds.flatMap fun id =>
    Cell256.all.map fun c => (id, c)

theorem allOperatorIds_length : allOperatorIds.length = 371 := by native_decide
theorem allOperatorCells_length : allOperatorCells.length = 94976 := by native_decide
theorem allOperatorCells_length_product :
    allOperatorCells.length = allOperatorIds.length * Cell256.all.length
```

当前总覆盖网格为:

```text
OperatorCell = OperatorId × Cell256
371 × 256 = 94976
```

(v1 之 71232 升级至 v3 之 94976; 增量 = 371 × 64 = 23744 cells)

读法:

| 层 | 问题 | 当前状态 |
|---|---|---|
| 覆盖 | 每个算子是否能配到每个格点 | `OperatorCellMap` 已覆盖 |
| 可执行 | 每个算子是否有保守机器 denotation | WenSurface / semantics 层已有分层支持 |
| 精确变换 | 是否能证明为 `Cell256 -> Cell256` 的具体变换 | 只在专门语义文件中对部分算子闭合 |
| 领域专义 | 是否解释完每个字在每个领域中的完整意义 | 未完成, 且不应由覆盖网格冒充 |

因此, `371 × 256` 的拓扑不是 "94976 个孤立事实", 而是一个二部索引关系:

```text
OperatorId  ----作用/读法/候选语义---->  Cell256
```

把它看成图, 可以有三种边:

| 边 | 来源 | 含义 |
|---|---|---|
| 格点边 | `flip1..flip6`、`hexCuo / hexZong / hexHu`、`shiCuo / shiZong / shiCuoZong`、`yin / tou` | `Cell256` 内部可达 (8 atomic XOR + 6 outer permutations) |
| 算子边 | `OperatorId` 的语义 / signature | 算子如何作用或约束格点 |
| 名指边 | `Roster.Symbol` / surface reading | 名如何指向算子、物、间或缘 |

第三种边正是当前最缺的桥.

## 六 · 从算到史 (v3 升级版)

单个算子给出一步作用; 算子列给出路径:

```text
算子        : Cell256 -> Cell256        -- 精确情形
算子列      : Cell256 -> Cell256        -- 复合
路径        : c0 -> c1 -> ... -> cn
史          : List Cell256
```

现有 `史带结构` 已把 `YiState.history : List Cell256` 读成史带. 本文补充的是:

```text
格受算, 算成径, 径积为史.
```

这里的史不是单纯时间顺序, 而是 "格点经过算子作用后留下的可审计轨迹". 因此史至少包含三类信息:

| 信息 | 说明 |
|---|---|
| 起点 | 从哪个 `Cell256` 开始 |
| 算子列 | 经由哪些作用、判断、跳转、推取 |
| 留痕 | 哪些格点被写入 `history` |

再往上, 律不是新格点, 而是路径之间的关系:

```text
两条路径是否同端点
两个算子是否交换
某个算子是否自反 / involutive
某个循环是否回到原点
某个程序是否停机
```

这些律能被形式化时, 才进入证.

> **v3 升级**: BaguaTuring (= YiInstr + YiState + partial def run) 之 `history : List Cell256` 是史带; 主程序之 `runFuel n` 之 trace 是路径. Halting predicate `Halts P h := ∃ n, runFuel n halts` 是「径之 final state 是否落入 halt set」之 predicate. Halting 不可判 (per [J](J_理之不完备_哥德尔在256.md)) = 无 r.e. 判机能从「算子列 + 起点」均一判定「径是否归于 halt」.

```text
NameTargetKind := thing | interval | relation

  name       : Symbol
  targetKind : NameTargetKind
  surface    : String
  reading?   : Option OperatorReading

```

第一批覆盖不追求全 371, 只覆盖构名核心:


验收口径应保守:

| 阶段 | 目标 |
|---|---|

这样做的好处是边界清楚: 名册不冒充语义执行器, 算子目录不冒充对象本体, `256 × 371` 不冒充完整解释学. 三者通过一层可审计对齐表相接.
