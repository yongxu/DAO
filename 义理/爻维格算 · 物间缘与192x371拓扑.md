## 〇 · 边界声明

本文不主张：

| 不主张 | 原因 |
|---|---|
| `192 × 371` 是一个 71232 维空间 | 它是算子与格点的覆盖网格，不是坐标维度 |
| 每个 `OperatorId × Cell192` 都已有完整领域专义 | 当前有覆盖、可执行与若干精确语义分层，不等于全部文本专义闭合 |
| `Cell192` 已被 Lean 命名为 `Q6 □ C3` | `Q6 □ C3` 是本文的拓扑读法；Lean 只证明 `Bool^6 × Shi` 与相关边操作 |
| `物间事` 被废弃 | 本文只把 `事` 下读为依 `物 / 间 / 缘` 而成的显现 |
| 名册已经完全对齐算子 | 当前正缺一层 `NameOperatorAlignment` |

本文主张：

| 主张 | 精确读法 |
|---|---|
| 爻生维 | 一个爻位是一条二值维度；一个爻值是该维度上的点 |
| 维成格 | 六个爻位给出 `Bool^6`，即六十四卦格点 |
| 格加时成 192 | `Cell192 = Hexagram × Shi = Bool^6 × Z/3` |
| 格受算 | `OperatorId` 不是维度，而是作用在格点上的可执行 / 可索引算子目录 |
| 算成径 | 算子列给出路径、变换、程序或关系链 |
| 径积为史 | `YiState.history : List Cell192` 是路径留下的带 |
| 史中出律 | 路径之间可比较、归约、证明，形成律 |

## 一 · 爻到维

爻不是普通对象，而是最小可分差异。它有两种读法：

| 读法 | 形式 |
|---|---|
| 爻位 | 一条可独立翻转 / 判别的二值维度 |
| 爻值 | 该维度上的一个点：阴或阳 |

因此：

```text
一个爻位 = 一维 Bool
一个爻值 = Bool 中的一个点
n 个爻位 = Bool^n = {阴, 阳}^n
```

层级展开为：

| 爻位数 | 结构 | 点数 | 易层读法 |
|---:|---|---:|---|
| 0 | `Unit` | 1 | 一点 / 未分 |
| 1 | `Bool` | 2 | 一爻 / 阴阳 |
| 2 | `Bool^2` | 4 | 四象 |
| 3 | `Bool^3` | 8 | 八卦 |
| 6 | `Bool^6` | 64 | 六十四卦 |

所以“从爻到维”的关键不是“爻等于 0 维”，而是：

```text
爻位产生维度；爻值落成点。
```

这也解释为什么三爻自然是八卦，六爻自然是六十四卦：每加一个可独立判别的爻位，就把状态空间乘上一个 `Bool`。

## 二 · 物间缘

本文把更底层的成事结构写为：

```text
物 × 间 × 缘 -> 事
```

其中：

| 字 | 读法 |
|---|---|
| 物 | 可被指认、可承载同一性的对象面 |
| 间 | 可定位、分辨、相邻、先后、内外的关系场 |
| 缘 | 使物与间发生牵连并可能成事的条件关系 |
| 事 | 物在间中依缘而成的一次显现 |

这比直接说 `物 × 间 × 事` 更细，因为 `事` 不再是根槽位，而是结果层。现有 `物间事三位一体` 文档仍可保留为接口读法：

```text
WuJianShiExpression Thing Interval Event
```

在本文中，它被解释为：

```text
Thing    -> 物
Interval -> 间
Event    -> 物间缘合成后的事
```

所以旧接口没有被否定；只是 `缘` 补上了“为什么能成事”的条件层。

## 三 · 名是指针

名不是对象本身。名更像一个指针、handle 或可审计入口：

```text
名 : 爻 -> Target
Target = 物 | 间 | 缘
```

名可以指向物、间或缘；类则是名的分束，不是简单 alias。

```text
类 = 按同指、同谓、同缘，把名分束
```

构名时，许多单字不是被命名的对象，而是构名算子：

| 字 | 构名作用 | 例 |
|---|---|---|
| 者 | 动作的主动承载位 | `持者`、`出者` |
| 所 | 动作的被指 / 被偶位 | `所指`、`所偶` |
| 之 | 属格 / 投影 / 应用 | `爻之名` |
| 为 | 视角转换 / ascription | `爻为物`、`爻为缘` |
| 主 | 单主面 | `主面` |
| 诸 | 多面 / 多系 / 多修 | `诸面`、`诸系`、`诸修` |
| 于 | 时位锚定 | `生于`、`起于` |
| 是 | 谓词化 / 判为某类 | `是物`、`是缘` |

这给出一组待对齐的核心构式：

```text
名        <- 爻
类        <- 别·名 / 分·名
主面      <- 主·面
诸面      <- 诸·面
生于      <- 生·于
焚于      <- 焚·于
起于      <- 起·于
寝于      <- 寝·于
持者      <- 持·者
出者      <- 出·者
所指      <- 所·指
所偶      <- 所·偶
诸系      <- 诸·系
诸修      <- 诸·修
见度      <- 见·度
爻之名    <- 爻·之·名
爻为物    <- 爻·为·物
爻为间    <- 爻·为·间
爻为缘    <- 爻·为·缘
```

其中 `爻为者 / 爻为持 / 爻为物 / 爻为间 / 爻为缘 / 爻为期` 不是六个实体列表，而是同一爻在不同目标类型下的视角化。

## 四 · 192 的拓扑

Lean 中：

```text
Cell192 = Hexagram × Shi
Hexagram = Bool^6
Shi = {已, 今, 未}
```

所以：

```text
Cell192 = Bool^6 × Z/3
```

这个式子有两个方向：

| 方向 | Lean 对象 | 拓扑读法 |
|---|---|---|
| 六爻方向 | `flip1` 至 `flip6` | 每个时态层内是一张 6-cube |
| 时态方向 | `shiNext`、`shiPrev` | 每个卦点上挂一个三时循环 |

若只看六爻翻转边，`Cell192` 分成三个互不相通的 64 点 6-cube：

```text
Q6 × {已}
Q6 × {今}
Q6 × {未}
```

加入 `shiNext / shiPrev` 后，每个卦点上的三时态连成一个循环：

```text
已 -> 今 -> 未 -> 已
```

于是本文可把 192 的图读作：

```text
Q6 □ C3
```

也就是六维二值立方体与三时循环图的 Cartesian product。这里的 `□` 是本文的图论读法，不是当前 Lean 中已有的命名定理。

还要区分另一个容易混淆的 192：

| 192 来源 | 公式 | 含义 |
|---|---|---|
| `Cell192` | `64 × 3` | 六十四卦乘三时态 |
| 6-cube 边数 | `6 × 2^5` | 六维立方体的边数 |

二者数值相同，但来源不同。前者是状态格点，后者是几何边数。

## 五 · 371 的位置

`371` 是文言算子目录的行数：

```text
allOperatorIds.length = 371
```

它不是第 371 个维度，也不是 371 个状态点。它更像一套可作用、可读、可索引的操作表。

当前总覆盖网格为：

```text
OperatorCell = OperatorId × Cell192
371 × 192 = 71232
```

读法：

| 层 | 问题 | 当前状态 |
|---|---|---|
| 覆盖 | 每个算子是否能配到每个格点 | `OperatorCellMap` 已覆盖 |
| 可执行 | 每个算子是否有保守机器 denotation | WenSurface / semantics 层已有分层支持 |
| 精确变换 | 是否能证明为 `Cell192 -> Cell192` 的具体变换 | 只在专门语义文件中对部分算子闭合 |
| 领域专义 | 是否解释完每个字在每个领域中的完整意义 | 未完成，且不应由覆盖网格冒充 |

因此，`371 × 192` 的拓扑不是“71232 个孤立事实”，而是一个二部索引关系：

```text
OperatorId  ----作用/读法/候选语义---->  Cell192
```

把它看成图，可以有三种边：

| 边 | 来源 | 含义 |
|---|---|---|
| 格点边 | `flip1..flip6`、`shiNext/shiPrev` | `Cell192` 内部可达 |
| 算子边 | `OperatorId` 的语义 / signature | 算子如何作用或约束格点 |
| 名指边 | `Roster.Symbol` / surface reading | 名如何指向算子、物、间或缘 |

第三种边正是当前最缺的桥。

## 六 · 从算到史

单个算子给出一步作用；算子列给出路径：

```text
算子        : Cell192 -> Cell192        -- 精确情形
算子列      : Cell192 -> Cell192        -- 复合
路径        : c0 -> c1 -> ... -> cn
史          : List Cell192
```

现有 `史带结构` 已把 `YiState.history : List Cell192` 读成史带。本文补充的是：

```text
格受算，算成径，径积为史。
```

这里的史不是单纯时间顺序，而是“格点经过算子作用后留下的可审计轨迹”。因此史至少包含三类信息：

| 信息 | 说明 |
|---|---|
| 起点 | 从哪个 `Cell192` 开始 |
| 算子列 | 经由哪些作用、判断、跳转、推取 |
| 留痕 | 哪些格点被写入 `history` |

再往上，律不是新格点，而是路径之间的关系：

```text
两条路径是否同端点
两个算子是否交换
某个算子是否自反 / involutive
某个循环是否回到原点
某个程序是否停机
```

这些律能被形式化时，才进入证。

## 七 · NameOperatorAlignment 路线图

当前缺口可以命名为：

```text
NameOperatorAlignment
```

目标不是再造一个大词表，而是把三类已有事实接起来：

| 已有层 | 代表对象 | 缺口 |
|---|---|---|
| 名册 | `Roster.Symbol`、`AtomName`、`GenName` | 名的根、依赖、分类已有 |
| 读法 | `OperatorReading`、`SurfaceReadings` | 字形到算子读法已有一部分 |
| 算子 | `OperatorId`、`OperatorCell` | 371 与 192 覆盖已有 |

最小新接口可先只做 typed skeleton：

```text
NameTargetKind := thing | interval | relation
NameOperatorAlignment :=
  name       : Symbol
  targetKind : NameTargetKind
  surface    : String
  reading?   : Option OperatorReading
  operator?  : Option OperatorId
```

第一批覆盖不追求全 371，只覆盖构名核心：

```text
之、者、所、于、是、为、名、类、持、指、偶、缘
```

验收口径应保守：

| 阶段 | 目标 |
|---|---|
| A | 每个核心构名字在名册中可定位 |
| B | 能列出该字已有 `OperatorReading` 或明确 pending |
| C | 能说明它指向 `物 / 间 / 缘` 哪类目标 |
| D | 能解释构式如何生成 `爻之名`、`爻为缘`、`所指` 等复合名 |
| E | 再考虑是否接入 `OperatorCellMap` 的 192 × 371 网格 |

这样做的好处是边界清楚：名册不冒充语义执行器，算子目录不冒充对象本体，`192 × 371` 不冒充完整解释学。三者通过一层可审计对齐表相接。
